üì¶ Encapsulation in OOP

Encapsulation is the practice of separating an object's internal implementation from its external interface. This ensures that internal details are hidden and protected from external access, promoting modularity and maintainability.

üîì Public Properties

Public properties are accessible from anywhere. They form the external interface of a class and can be freely read or modified. 

class CoffeeMachine {
  waterAmount = 0; // public
}

üõ°Ô∏è Protected Properties (Convention)

JavaScript does not natively support protected properties. However, by convention, properties prefixed with an underscore (_) are treated as protected, indicating they should not be accessed outside the class or its subclasses. 

class CoffeeMachine {
  _waterAmount = 0;

  set waterAmount(value) {
    if (value < 0) throw new Error("Negative water");
    this._waterAmount = value;
  }

  get waterAmount() {
    return this._waterAmount;
  }
}

üîê Private Properties (#)

Private properties are denoted by a hash (#) and are only accessible within the class they are defined. This is enforced by the language, providing true encapsulation. 

class CoffeeMachine {
  #waterLimit = 200;

  #fixWaterAmount(value) {
    if (value < 0) return 0;
    if (value > this.#waterLimit) return this.#waterLimit;
    return value;
  }

  setWaterAmount(value) {
    this.#waterLimit = this.#fixWaterAmount(value);
  }
}

Attempting to access #waterLimit or #fixWaterAmount from outside the class will result in an error.

‚ö†Ô∏è Limitation in Inheritance

Private fields are not accessible in subclasses. Each class has its own separate set of private fields, and attempting to access a superclass's private field in a subclass will result in a syntax error. 

class Parent {
  #secret = 'hidden';
}

class Child extends Parent {
  revealSecret() {
    console.log(this.#secret); // SyntaxError
  }
}

This behavior enforces strict encapsulation but limits the ability to share internal state between parent and child classes. If shared access is needed, consider using protected-like conventions or public methods to expose necessary functionality.

üìå Summary

  • Public: Accessible from anywhere.
  • Protected (_): Conventionally treated as protected; accessible within the class and subclasses.
  • Private (#): Truly private; accessible only within the class.

Encapsulation helps in hiding complexity, protecting internal states, and creating a clear separation between internal and external interfaces.

üë®‚Äçüíº Notes for Senior/Lead Developers

  • Encourage the use of private fields (#) for true encapsulation, especially in large codebases.
  • Use protected conventions (_) judiciously, understanding they are not enforced by the language.
  • Be cautious with inheritance; private fields are not accessible in subclasses.
  • Ensure code consistency by following established naming conventions and access patterns.
  • Stay updated with browser support for private fields and consider transpilation if necessary.
