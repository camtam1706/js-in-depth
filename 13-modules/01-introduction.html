üß© 1. What is a Module?

  • A module is just a file. One script is one module.
  • Modules can import functions, objects, or primitives exported from other modules, and export their own.
  • Modules always use strict mode by default.
  • Each module has its own top-level scope. 
// lib.js
export let sayHi = function(user) {
  alert(`Hello, ${user}!`);
};

// main.js
import { sayHi } from './lib.js';
sayHi('John'); // Hello, John!

‚ú® 2. Module Features

  • Always strict mode: Modules always behave as if 'use strict' is present.
  • Own scope: Variables and functions are only visible inside the module unless explicitly exported.
  • Module code only executes once: If a module is imported multiple times, its code runs only the first time, and then exports are reused.
  • Top-level this is undefined: In modules, this at the top level is undefined.

üì• 3. Import / Export

  • Use export to make variables, functions, or classes available outside the module.
  • Use import to access exported entities from other modules.
  • Named exports: Export multiple named items from a module. 
    export { sayHi, sayBye };
import { sayHi, sayBye } from './lib.js';
  • Default export: A module can have a single default export. 
    export default function() { ... }
import myFunc from './lib.js';

üåê 4. Module Scripts in HTML

  • To load a module, use <script type="module"> in HTML.
  • Modules are deferred by default (like defer).
  • Module scripts support import/export syntax. 
<script type="module">
  import { sayHi } from './lib.js';
  sayHi('John');
</script>

üîÑ 5. Module Loading Characteristics

  • Module scripts run after the HTML is loaded and processed.
  • External module scripts are loaded with CORS (cross-origin requests allowed by default).
  • Same-origin policy: Local modules work via HTTP/HTTPS, not via file:// directly (security restrictions).
  • async and defer attributes have no effect on module scripts‚Äîthey always behave as deferred.

‚ö° 6. Differences from Non-Module Scripts

  • Module scripts are always deferred; regular scripts can be blocking or deferred.
  • Variables/functions in modules are not added to the global scope.
  • Module scripts are loaded with CORS; non-module scripts are not.

üåç 7. Browser Compatibility & Usage

  • All major modern browsers support modules.
  • For older browsers, modules can be transpiled using tools like Babel/Webpack.
  • Note: Local module scripts may not work directly from the filesystem. Use a server (like localhost or live-server).

üöÄ 8. Senior/Lead: What You Should Watch Out For

  • Design modular code: Keep modules focused, single-responsibility, and avoid circular dependencies.
  • Leverage tree-shaking: Use ES modules to allow bundlers to remove unused code for smaller builds.
  • Scope management: Avoid polluting the global scope. Encapsulate logic within modules.
  • Interoperability: Understand differences between ES modules and CommonJS (Node.js). For cross-environment code, choose module systems carefully.
  • Import strategies: Prefer explicit imports over import * to keep dependencies clear and facilitate tree-shaking.
  • Performance: Minimize the number of imports/exports in hot code paths to reduce overhead.
  • Security: Mind CORS and avoid exposing sensitive logic via public modules.
  • Testing: Design modules to be testable, with clear interfaces and minimal side effects.
  • Deployment: Ensure all modules are bundled/minified as appropriate for production; beware of path/case sensitivity issues across OS.

üìù 9. Summary

  • JavaScript modules are essential for scalable, maintainable codebases.
  • Modules provide encapsulation, reuse, and better dependency management.
  • Understanding both the syntax and the underlying browser mechanics is key for advanced usage and troubleshooting.
