üìù Introduction to Modules

  • Modules are separate files that export and import functionality to organize code and reuse components.
  • Each module is executed only once; its exported variables and functions are shared.
  • Modules always use strict mode by default.

üì§ Exporting in JavaScript Modules

  • Named Exports: Export multiple things per module (variables, functions, classes). 
    export let name = "John";
export function sayHi(user) { alert(`Hello, ${user}!`); }
export class User { /* ... */ }
  • Export List: Export previously declared items. 
    function sayBye(user) { alert(`Bye, ${user}!`); }
export { sayBye };
  • Default Export: Export a single main value from the module. 
    export default function() { ... }
export default class { ... }
export default value;

üì• Importing from Modules

  • Importing Named Exports: 
    import { sayHi, sayBye } from './say.js';
sayHi('John');
  • Renaming Imports: 
    import { sayHi as hi, sayBye as bye } from './say.js';
  • Import Everything (Namespace): 
    import * as say from './say.js';
say.sayHi('John');
  • Import Default Export: 
    import User from './user.js';
  • Import Default + Named: 
    import User, { sayHi } from './user.js';
  • Import Without Assigning (run module code only): 
    import './module.js';

üß© Module Features & Restrictions

  • Imports and exports must be at the top-level (not inside functions, conditions, or blocks).
  • Imports are static, resolved at compile time, not dynamically.
  • The imported bindings are live-connected to the original values.
  • Module code runs only once, even if imported multiple times.
  • File Extension: Usually must include the file extension (.js).
  • Type="module": Script tags or bundlers must indicate that files are ES modules. 
    <script type="module" src="main.js"></script>
  • Modules are deferred by default (like defer attribute).
  • Module scripts have their own scope (variables don‚Äôt leak to global).

‚ö° Dynamic Imports

  • You can import modules dynamically using import() as a function, returning a promise. 
    let module = await import('./path/to/module.js');
module.hello();
  • Dynamic import enables loading code on demand, conditional loading, or loading modules by variable path.

‚ö†Ô∏è Common Pitfalls & Gotchas

  • Top-level Only: All import/export statements must be at the root, not inside if, for, or functions.
  • Cannot Import Conditionally: Use import() for conditional/dynamic loading.
  • Circular Dependencies: Modules can import each other, but only what‚Äôs been exported so far is available (undefined for later exports).
  • Extension Required: When importing, specify the exact file extension in browsers (./file.js, not ./file).

üöÄ Senior/Lead Considerations & Best Practices

  • Design for Maintainability: Use named exports for clarity; default exports are useful for one-main-purpose modules. Avoid mixing too many exports in a single file.
  • Structure Modules Logically: Organize codebase by domain, feature, or responsibility. Keep modules small, focused, and reusable.
  • Leverage Dynamic Imports: For code splitting, lazy loading, and optimizing initial load time (especially in large SPA projects).
  • Handle Circular Dependencies: Refactor or use intermediate modules to break cycles. Keep dependency graph as flat as possible.
  • Code Review Focus: Ensure that module boundaries are respected, exports/imports are clean, and there are no ‚Äúhidden‚Äù side effects.
  • TypeScript/ESLint: Use tools for enforcing import/export style, module boundaries, and to catch accidental mistakes or import cycles.
  • Tree Shaking: Prefer named exports for better tree-shaking with modern bundlers.
  • Explicit File Extensions: Always specify extensions to avoid runtime issues, especially in ESM-aware environments and browsers.
  • Secure Imports: Avoid importing from untrusted sources, and watch out for supply chain attacks.
  • Documentation: Document module APIs and usage, especially for shared utility or core business logic modules.
