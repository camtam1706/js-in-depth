üìë Node Types Overview

  • The DOM represents every part of an HTML document as a node. Main node types:
    • Element: represents an HTML element, e.g., <body> (nodeType == 1)
    • Text: represents text inside elements (nodeType == 3)
    • Comment: represents a comment (nodeType == 8)
    • Other: document, documentType, processingInstruction (rarely used in everyday work)
  • All nodes inherit from the base Node class.

üõ†Ô∏è nodeType, nodeName, and tagName

  • nodeType: a numeric code for node type. E.g., 1 for Element, 3 for Text, 8 for Comment.
  • nodeName (for all nodes): Returns the name of the node, e.g., 'DIV', '#text', '#comment'.
  • tagName (only for Element nodes): Returns the element's tag in uppercase, e.g., 'DIV'.
  • For elements, nodeName === tagName.
  • Best practice: Prefer nodeType for type checks (since nodeName can be different for XML/XHTML).
Example:
alert( document.body.nodeType ); // 1
alert( document.body.nodeName ); // BODY

üìù innerHTML, outerHTML, and nodeValue/data

  • innerHTML: Gets/sets the HTML inside an element (string).
  • outerHTML: Gets/sets the HTML of the element itself (string). Assigning to outerHTML replaces the element.
  • nodeValue/data:
    • For Text or Comment nodes, nodeValue or data gives the text content.
    • For Element nodes, nodeValue is always null.
Example:
let comment = document.createComment("Hello");
alert(comment.nodeValue); // "Hello"
alert(comment.data); // "Hello"

üîÑ textContent and hidden Properties

  • textContent: Gets/sets the text inside the node and all descendants.
    • Ignores any HTML tags (treats everything as plain text).
    • Setting textContent removes all child nodes and replaces them with a single text node.
  • hidden: Boolean property. If set to true, the element is not visible (CSS display: none).
Example:
document.body.textContent = "Hello!"; // replaces all HTML with plain text "Hello!"

üßÆ Special Properties for Different Node Types

  • Element: has tagName, innerHTML, outerHTML, hidden, textContent, and more.
  • Text/Comment: has nodeValue and data for content.
  • Document: has special properties for the root of the DOM tree.
  • Best Practice: Check nodeType before accessing these properties.

‚ö° Senior/Lead: Key Takeaways & Pitfalls

  • Performance: Frequent use of innerHTML or outerHTML for updates can cause reflows and repainting. Prefer DOM methods (like appendChild, removeChild, etc.) for performance-sensitive code.
  • Security: Assigning unsanitized HTML to innerHTML/outerHTML can open the door for XSS attacks. Always sanitize or validate content.
  • NodeType Checks: Rely on nodeType for robust type checking across XML/HTML/XHTML.
  • Replacing vs. Updating: Replacing elements via outerHTML removes all event listeners and data associated with the original node. Use with caution.
  • textContent vs. innerHTML: textContent is safer for inserting plain text, as it does not parse HTML.
  • Hidden Property: Using hidden is cleaner than toggling display styles directly.
  • Debugging: Be aware that nodeName may vary in XML vs. HTML documents (case sensitivity).
  • Team Onboarding: Educate team members on the risks and differences between manipulating via innerHTML, textContent, and direct DOM APIs.
