Window Sizes: innerWidth, innerHeight, outerWidth, outerHeight

  • window.innerWidth & window.innerHeight: Viewport width/height (visible page area, excluding toolbars, etc.)
  • window.outerWidth & window.outerHeight: Total browser window size, including all interface elements (toolbars, tabs, etc.)
  • Useful for responsive layouts and detecting viewport changes.
Example: 
console.log(window.innerWidth);  // e.g., 1200
console.log(window.innerHeight); // e.g., 700

Scrolling: scrollX, scrollY, scrollTo, scrollBy, scroll

  • window.scrollX & window.scrollY: Current horizontal and vertical scroll offsets in pixels.
  • window.scrollTo(x, y): Scrolls window to specific coordinates.
  • window.scrollBy(dx, dy): Scrolls by the given amount from current position.
  • window.scroll(options): Same as scrollTo but accepts an options object for smooth scrolling: 
    window.scrollTo({
  top: 100,
  left: 0,
  behavior: "smooth"
});
Tip: Use behavior: "smooth" for smooth scrolling UX.

Document Size: scrollWidth, scrollHeight, clientWidth, clientHeight

  • document.documentElement.scrollWidth / scrollHeight: Full width/height of content, including invisible (scrolled out) area.
  • document.documentElement.clientWidth / clientHeight: Viewport width/height for the document (excluding scrollbars, borders).
  • Usually used to detect how much content is available and whether scrolling is possible.
  • For <body> and other elements, similar properties apply.
Example: 
let scrollHeight = document.documentElement.scrollHeight;
let clientHeight = document.documentElement.clientHeight;
let scrollable = scrollHeight > clientHeight;

Scroll Detection: Checking if Scrolled to Bottom

  • To check if user scrolled to the bottom:
    window.scrollY + window.innerHeight ≥ document.documentElement.scrollHeight
  • This is useful for infinite scroll, lazy loading, etc.
if (window.scrollY + window.innerHeight >= document.documentElement.scrollHeight) {
  // User has scrolled to the bottom
}

Mobile & Cross-browser Considerations

  • Scrollbars may have zero width on mobile and some browsers.
  • Scroll-related properties (scrollX, scrollY) may behave differently across devices.
  • Always test your code on both desktop and mobile browsers for consistent results.

Common Patterns & Gotchas

  • Scroll position can be negative in RTL (right-to-left) layouts in some browsers.
  • Some properties may refer to document.body or document.documentElement depending on the browser (especially IE).
  • Always use document.documentElement for most reliable results.
  • On mobile, the address bar may appear/disappear, changing viewport height dynamically.

Senior/Lead Notes: Architecture & Best Practices

  • Debounce scroll events: Frequent scroll/resize events can cause performance issues. Always debounce or throttle listeners when tracking scroll or resize.
  • Prefer Intersection Observer: For complex scroll detection (e.g., infinite scroll, lazy loading), prefer IntersectionObserver API over manual scroll position checks.
  • Handle dynamic layouts: Remember that page content and viewport size may change after initial load (dynamic content, UI expansion, mobile browser chrome behavior).
  • Accessibility: Ensure keyboard navigation and focus management work correctly when implementing scroll logic (e.g., don't steal focus unexpectedly).
  • Test on all environments: Differences in browser/OS, especially mobile, can cause unexpected scroll/size issues. Always test scroll/size logic under real device conditions.
  • Consider virtual scrolling: For large lists, consider virtualization libraries to improve performance and UX.
  • Custom Scrollbars: Custom-styled scrollbars may break native behaviors. Weigh UX gains vs. accessibility and performance tradeoffs.
